Executive Summary

On February 25, 2026, SOC Analyst Vishvdeep Bal conducted a structured threat hunt on the Log(N) Pacific Cyber Range, identifying and fully reconstructing a sophisticated Adversary-in-the-Middle (AiTM) attack against the Microsoft 365 account of m.smith@lognpacific.org.

The intrusion was attributed to Scattered Spider (UNC3944 / Octo Tempest), a financially motivated threat group previously linked to breaches at MGM Resorts and Caesars Entertainment. The group is known for Infostealer credential harvesting, MFA fatigue push harassment, AiTM reverse-proxy phishing, and Business Email Compromise fraud.

⚠ Critical Defensive Gap

ConditionalAccessStatus was notApplied on every attacker sign-in. A single Conditional Access policy requiring compliant, managed devices would have blocked the attacker's Linux/Firefox 147.0 session replay entirely — regardless of whether the session token was valid.

The attacker obtained Mark Smith's credentials via Infostealer malware, then used a reverse-proxy phishing framework (consistent with Evilginx) to intercept his session token mid-authentication, fully bypassing MFA. Operating from 205.147.16.190 (Netherlands), the attacker achieved account takeover at 21:59:52 UTC after 3 pre-success attempts, then executed a multi-stage post-compromise campaign.

Hunt Result

28 of 29 challenges solved · 2410 points · 71st place globally on the Log(N) Pacific leaderboard. The complete attack was correlated through a single AAD Session ID, demonstrating the forensic value of session-level correlation across SigninLogs, CloudAppEvents, and EmailEvents in Microsoft Sentinel.

Hunt Challenge Evidence — Log(N) Pacific Cyber Range

All 28 challenges below were solved using KQL against the LAW-Cyber-Range Log Analytics Workspace in Microsoft Sentinel. Each entry shows the challenge question, the answer confirmed correct, and the analytical reasoning.

28 challenges · 2410 pts total
Q02 Attacker Source IP 50 pts ✓ CORRECT ▶
✓ 205.147.16.190

Baselining Mark's daytime IP (172.175.65.103) against evening sessions revealed 205.147.16.190 as the anomalous IP. SigninLogs filtered to Mark's UPN and compared IPAddress values between daytime and evening sessions isolated the attacker's Netherlands-geolocated IP.

Q03 Attack Origin Country 50 pts ✓ CORRECT ▶
✓ NE (Netherlands)

Location field in SigninLogs confirmed country code NE — Netherlands. An employee authenticating from the Netherlands during off-hours from a different device profile is a clear geographic anomaly signal.

Q04 MFA Denial Error Code 75 pts ✓ CORRECT ▶
✓ 50074

ResultType 50074 in SigninLogs confirmed the attacker hit the MFA gate — "Strong Authentication is required." This error code indicates MFA was prompted but not completed, distinguishing it from a KMSI/session interrupt (50140).

KQL — MFA Error Detection
SigninLogs | where IPAddress == "205.147.16.190" | where UserPrincipalName == "m.smith@lognpacific.org" | project TimeGenerated, ResultType, ResultDescription | sort by TimeGenerated asc
Q05 MFA Fatigue Intensity 75 pts ✓ CORRECT ▶
✓ 3

KQL count of pre-success non-zero ResultType entries from the attacker's IP returned exactly 3: one 50074 (MFA required) and two 50140 (KMSI interrupts). This is the MFA fatigue count — the number of times the attacker was blocked before the token replay succeeded.

Q06 Application Accessed Post-MFA 75 pts ✓ CORRECT ▶
✓ One Outlook Web

SigninLogs sorted ascending after the first successful attacker sign-in revealed 'One Outlook Web' as the very first application accessed. A remote attacker without the desktop app installed would use the web version, confirming non-managed device access.

Q07 Attacker Operating System 75 pts ✓ CORRECT ▶
✓ Linux

DeviceDetail JSON field in SigninLogs confirmed Linux OS. Mark's corporate device runs Windows 10 — so Linux is a clear anomaly. Three anomaly layers: different country + different OS + different browser, all deviating from Mark's established baseline.

KQL — DeviceDetail OS Extraction
SigninLogs | where IPAddress == "205.147.16.190" | project TimeGenerated, AppDisplayName, DeviceDetail // DeviceDetail shows: {"operatingSystem":"Linux","browser":"Firefox 147.0","isCompliant":false,"isManaged":false}
Q08 Attacker Browser 75 pts ✓ CORRECT ▶
✓ Firefox 147.0

DeviceDetail browser field confirmed Firefox 147.0 — vs Mark's Edge 145.0 baseline. Cross-referencing with Mark's normal browser completes the 3-layer anomaly profile: different country (NL), different OS (Linux), different browser (Firefox vs Edge).

Q09 First Post-Auth Action 100 pts ✓ CORRECT ▶
✓ MailItemsAccessed

CloudAppEvents sorted ascending from the attacker's IP — MailItemsAccessed was the very first post-auth action. This is classic attacker behaviour: read the inbox first to understand the target, identify ongoing financial conversations, and plan the BEC lure before setting up persistence.

Q10 Persistence Rule Creation Method 75 pts ✓ CORRECT ▶
✓ New-InboxRule

ActionType field in CloudAppEvents confirmed New-InboxRule as the persistence mechanism. Inbox rules are silent, persistent, and often overlooked by defenders — making them a favourite Scattered Spider technique for maintaining data access and suppressing security alerts.

Q11 Forward Rule Name 100 pts ✓ CORRECT ▶
✓ . (single period)

RawEventData Parameters Name field revealed '.' (single period) — nearly invisible in any inbox rules UI. Parsing the RawEventData JSON from the New-InboxRule CloudAppEvent and expanding the Parameters array for a Property.Name == "Name" field revealed the covert rule name.

KQL — Extract Inbox Rule Name
CloudAppEvents | where IPAddress == "205.147.16.190" | where ActionType == "New-InboxRule" | extend DetailedData = parse_json(RawEventData) | mv-expand Property = DetailedData.Parameters | where Property.Name == "Name" | extend RuleName = tostring(Property.Value) | project TimeGenerated, AccountDisplayName, ActionType, RuleName, IPAddress
Q12 Forward Destination 75 pts ✓ CORRECT ▶
✓ insights@duck.com

ForwardTo parameter in the RawEventData Parameters array contained insights@duck.com — the attacker's external exfiltration address for financial emails. This is a critical IOC for email gateway blocking.

Q13 Forward Rule Keywords 100 pts ✓ CORRECT ▶
✓ invoice, payment, wire, transfer

SubjectOrBodyContainsWords on rule '.' = invoice, payment, wire, transfer — confirming BEC and invoice fraud intent. The attacker only wants financial emails forwarded; other email traffic is irrelevant to their objective.

Q14 Rule Processing Flag 100 pts ✓ CORRECT ▶
✓ StopProcessingRules

StopProcessingRules:True on both rules prevents Mark's own alert rules from ever firing. Smart attackers configure this parameter so no other rules process the matched emails after theirs — ensuring the user's own defensive rules are silenced.

Q15 Delete Rule Name 100 pts ✓ CORRECT ▶
✓ .. (double period)

The second inbox rule named '..' (double period) is set for deletion — not forwarding. Smart attackers don't just steal; they hide. The double-period name is designed to be nearly invisible alongside the single-period rule in any list view.

Q16 Delete Rule Keywords 100 pts ✓ CORRECT ▶
✓ suspicious, security, phishing, unusual, compromised, verify

Keywords on the delete rule '..': suspicious, security, phishing, unusual, compromised, verify — a complete security blackout. Any email from Microsoft, IT, or the SOC warning Mark about the compromise is automatically deleted before he sees it. Identity alerts, IT notifications, and self-correction warnings from colleagues are all suppressed.

Q17 BEC Target 75 pts ✓ CORRECT ▶
✓ j.reynolds@lognpacific.org

EmailEvents pivoted to the attacker IP revealed j.reynolds@lognpacific.org (Jay Reynolds, Finance) as the BEC target. The attacker profiled the inbox first to identify financial contacts before crafting the wire fraud request.

Q18 BEC Subject Line 75 pts ✓ CORRECT ▶
✓ RE: Invoice #INV-2026-0892 – Updated Banking Details

The attacker used classic thread hijacking — replying to an existing invoice thread rather than creating a new email. The 'RE:' prefix maximises credibility, making the recipient believe this is a continuation of a known conversation. The subject reveals the social engineering pretext: updated banking details for an existing invoice.

Q19 Email Direction 75 pts ✓ CORRECT ▶
✓ Intra-org

EmailDirection = Intra-org — the BEC email was sent internally, bypassing all external email gateway filters. An email from Mark Smith to Jay Reynolds appears as completely internal and legitimate, circumventing any external sender inspection rules.

Q20 BEC Sender IP 75 pts ✓ CORRECT ▶
✓ 205.147.16.190

SenderIPv4 = 205.147.16.190 — cross-referencing the sign-in IP with the email sender IP proves the same stolen session was used for both authentication and email sending. This is a definitive forensic link tying the BEC email to the attacker's session.

Q21 Cloud App Accessed (OneDrive) 75 pts ✓ CORRECT ▶
✓ Microsoft OneDrive for Business

CloudAppEvents FileAccessed action from the attacker IP confirmed Microsoft OneDrive for Business. Data exposure extends beyond email to personal cloud storage — the attacker scoped all available data stores beyond just the inbox.

Q22 SharePoint App Accessed 75 pts ✓ CORRECT ▶
✓ Microsoft SharePoint Online

SigninLogs confirmed Microsoft SharePoint Online — including PageViewed, ListCreated, ListColumnCreated, and breaking sharing inheritance. Data exposure spans both personal cloud storage (OneDrive) and corporate file shares (SharePoint), maximising the breach scope.

Q23 Session Correlation (AAD Session ID) 100 pts ✓ CORRECT ▶
✓ 00225cfa-a0ff-fb46-a079-5d152fcdf72a

AppAccessContext.AADSessionId in the inbox rule RawEventData was cross-referenced with the SessionId in SigninLogs. A single session ID ties all attacker activity: sign-in, inbox access, rule creation, file access, and BEC email sending — definitive forensic correlation across all log sources.

KQL — Session Correlation
SigninLogs | where TimeGenerated between (datetime(2026-02-25T21:00:00Z) .. datetime(2026-02-25T23:00:00Z)) | where IPAddress == "205.147.16.190" | where ResultType == 0 | project TimeGenerated, SessionId, AppDisplayName, ConditionalAccessStatus
Q24 Conditional Access Status 100 pts ✓ CORRECT ▶
✓ notApplied

ConditionalAccessStatus = notApplied on EVERY attacker sign-in — this is the critical defensive gap that enabled the entire attack. No Conditional Access policies were enforced against the attacker's sessions, meaning no check for compliant devices, managed devices, or geographic restrictions was ever applied.

Primary Gap — Immediate Fix Required

A single CA policy requiring managed/compliant devices blocks Linux/Firefox 147.0 access entirely, regardless of session token validity. This one control breaks the AiTM kill chain at the point of access.

Q25 MFA Fatigue MITRE ID 100 pts ✓ CORRECT ▶
✓ T1621

MFA Fatigue maps to MITRE ATT&CK T1621 — Multi-Factor Authentication Request Generation. This technique involves sending repeated MFA push notifications to wear down a user into approving the request, or as seen here, using it as cover while waiting for session token intercept via AiTM proxy.

Q27 Credential Source 100 pts ✓ CORRECT ▶
✓ Infostealer

Scattered Spider purchases credentials from Infostealer marketplace logs — malware that harvests saved passwords, session tokens, and browser data from infected machines. The attacker already had Mark's password before the MFA fatigue phase began, which is why they could immediately proceed to the AiTM phase once credentials were confirmed valid.

Q28 Immediate Containment Action 75 pts ✓ CORRECT ▶
✓ Revoke sessions

With the attacker holding a valid session token and both inbox rules still active, the single most critical immediate action is 'Revoke sessions' via Azure AD — this immediately invalidates the stolen session token and terminates all active attacker connections. All other remediation (password reset, rule deletion, IP blocking) follows but none is as time-critical as session revocation.

Q29 Threat Actor Attribution 100 pts ✓ CORRECT ▶
✓ Scattered Spider

All TTPs across the investigation confirm Scattered Spider (UNC3944): MFA fatigue harassment, AiTM session token theft, inbox rule persistence, BEC targeting finance, and use of anonymising infrastructure. The briefing mentioned a group that previously targeted MGM Resorts and Caesars Entertainment using identical techniques — Scattered Spider.

Conclusion

This threat hunt by Vishvdeep Bal on the Log(N) Pacific Cyber Range successfully reconstructed the complete Scattered Spider kill chain: from Infostealer credential theft, through AiTM MFA bypass, 30+ burst post-auth data accesses, dual inbox rule persistence, OneDrive/SharePoint exfiltration, and Business Email Compromise wire fraud. 28 of 29 challenges were solved, scoring 2410 points and placing 71st on the global leaderboard.

Most Actionable Finding

The single most actionable finding from this investigation is the ConditionalAccessStatus: notApplied gap. Enforcing a Conditional Access policy requiring compliant, managed devices would have blocked the attacker's Linux/Firefox 147.0 session replay entirely — regardless of whether their session token was valid. This one control breaks the AiTM kill chain at the point of access rather than at the MFA gate.

The complete attack was correlated through a single AAD Session ID (00225cfa-a0ff-fb46-a079-5d152fcdf72a), demonstrating the forensic value of session-level correlation across SigninLogs, CloudAppEvents, and EmailEvents in Microsoft Sentinel. KQL-based hunting in the LAW-Cyber-Range workspace provided definitive, evidence-grounded findings at every stage of the kill chain.

Key Detection Engineering Takeaway

Alert on New-InboxRule where the rule name is a single or double character and ForwardTo contains an external domain. Alert on successful sign-ins where ConditionalAccessStatus = notApplied from non-domestic IPs. These two detections would have surfaced this attack within minutes of the attacker's first successful authentication.
